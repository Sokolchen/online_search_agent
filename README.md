# online_search_agent

-本项目是基于[码士集团]教程，自行编写的一个使用LangChain作为基础架构的集成了基础功能的简单Agent。

-*其名为：Ken Masters！*-

# 已实现的功能

##1.基于智谱AI 联网搜索工具API实现的基础联网查找功能

-用户可对智能体直接对话要求查找"相关""最新"信息，引入`datetime`变量，确保查询时携带正确的时间信息，减少日期错乱现象，提升搜索结果的时效性。

##2.基于bs4与WebBaseLoader实现的简单推文网站爬取功能

-用户可提供指定网站并让智能体爬取内容，支持用户直接在对话中提出要解析的类名以提升精确度与效率，即按照自定义CSS类名的过滤。

##3.基于开源工具OpenDataLoaderPDFLoader与OpenAIEmbeddings实现的本地PDF文件解析+向量化存储的具有RAG问答管理功能的本地FAISS向量知识库存储功能

-管理查询功能

--对智能体提出“查询本地向量库/查询本地PDF存储”等文本，可查看本地向量管理情况与其对应PDF

-添加/删除功能

--在与智能体对话时，提出“添加PDF/向量文件”，在对话中给出适用于python（eg："E:/Re/online_search_agent/data/pdfs/CET_SET_RULES.pdf"）的路

径代码以添加新数据

--提出“删除PDF”并给出完整pdf名，将删除pdf对应的向量文件*注：本功能存在通过智能体调用无法成功运行的情况，正在解决中*

-对话查询知识功能

--当用户提出问题时，智能体首先尝试使用本地知识库工具回答问题，实现本地数据问答。

# 需要改良的地方

-当前程序仍存在一些需要优化的问题：

## 1. 尚未实现便捷可用的开发目标

-由于Web view知识的缺失，目前并未实现该agent在网页端的部署，需要依靠Langgraph本地服务器部署来实现调试。

## 2. 搜索引擎功能基础

-本项目采用的为智谱AI的基础版搜索引擎，搜索效果仅能实现基础功能。

## 3. 未在网页爬取功能中引入RAG功能来优化输出结构，在处理长文章时效果不佳

-如标题所示。

## 4. RAG功能架构较为基础简单，仅实现了基础功能

-文本切割与召回设置的效果尚未评估，但可以预见到的是，此基础架构将无法承担不断扩大的向量库规模下的工作。

-在加入文件越来越多时，不提升Top-K会导致返回信息不完整的严重问题，但这不可避免地提升运行时间与Token花费

-代码架构较为混乱，尚未整合相关函数以提升可读性与维护性

-尚未评估对Token的消耗情况与运行速度。

-可深入学习更多的RAG知识以搭建更好的架构
